Great question! Let me break it down step by step with real code examples.

The Problem First
Without fixtures, every test file looks like this:

typescript
// ❌ BAD - repeated in every spec file
test('user can search', async ({ page }) => {
  const homePage = new HomePage(page);
  await homePage.goto();
  await page.waitForLoadState('networkidle');
  // ... test logic
});
Fixtures solve this by injecting ready-to-use objects into tests automatically.

1. home.fixture.ts — Page Fixture
typescript
// src/fixtures/pages/home.fixture.ts

import { test as base } from '@playwright/test';
import { HomePage } from '../../pages/HomePage';

// Step 1: Define the "shape" of what this fixture provides
type HomeFixture = {
  homePage: HomePage;
};

// Step 2: Extend Playwright's base `test` with your custom fixture
export const test = base.extend<HomeFixture>({

  homePage: async ({ page }, use) => {
    // --- SETUP (before each test) ---
    const homePage = new HomePage(page);
    await homePage.goto();                      // navigates automatically
    await page.waitForLoadState('networkidle'); // waits until page is stable

    // --- HAND OVER to the test ---
    await use(homePage);                        // test runs here

    // --- TEARDOWN (after each test) ---
    // e.g. clear cookies, reset state if needed
  }

});
How a test uses it:

typescript
import { test } from '../fixtures/pages/home.fixture.ts';

// ✅ homePage is already navigated and ready — no setup needed
test('search works', async ({ homePage }) => {
  await homePage.searchFor('iPhone');
  await homePage.expectResultsVisible();
});

3. user.fixture.ts — Data Fixture
This one is the most powerful — it creates real data before the test and deletes it after.

typescript
// src/fixtures/data/user.fixture.ts

import { test as base } from '@playwright/test';
import { UserController } from '../../api/UserController';
import { UserFactory } from '../../../data/factories/UserFactory';

type UserFixture = {
  testUser: { email: string; password: string; id: string };
};

export const test = base.extend<UserFixture>({

  testUser: async ({ }, use) => {
    const userController = new UserController();
    
    // --- SETUP: Create a fresh user via API before test ---
    const userData = UserFactory.generate();       // { email: 'rand123@test.com', password: 'Abc@123' }
    const createdUser = await userController.createUser(userData);

    // --- HAND OVER to the test ---
    await use(createdUser);                        // test gets the real user object

    // --- TEARDOWN: Delete user after test finishes ---
    await userController.deleteUser(createdUser.id); // clean up, no leftover data
  }

});
How a test uses it:

typescript
test('user can login', async ({ page, testUser }) => {
  // testUser.email and testUser.password are real, freshly created credentials
  await page.fill('#email', testUser.email);
  await page.fill('#password', testUser.password);
  await page.click('#login-btn');
});
Every test run gets a brand new user, and it's deleted after. No test pollution.

4. page-factory.ts — The Merge Point
This is where all individual fixtures are combined into one single test object.

typescript
// src/fixtures/page-factory.ts

import { mergeTests } from '@playwright/test';

import { test as homeFixture }     from './pages/home.fixture';
import { test as checkoutFixture } from './pages/checkout.fixture';
import { test as navbarFixture }   from './components/navbar.fixture';
import { test as userFixture }     from './data/user.fixture';
import { test as authFixture }     from './auth/session.fixture';

// Merge ALL fixtures into one combined `test`
export const test = mergeTests(
  homeFixture,
  checkoutFixture,
  navbarFixture,
  userFixture,
  authFixture
);
5. index.ts — The Single Export
typescript
// src/fixtures/index.ts

export { test, expect } from './page-factory';
That's it. One line. Now every test file imports from one place.

.fixture.ts is just a naming convention — it's a regular TypeScript file
home.fixture.ts
home.page.ts
home.spec.ts
These are all just .ts files. The words fixture, page, spec before .ts are just naming conventions so developers instantly know what's inside the file. Playwright doesn't care about the name.

What Actually Matters is What's Inside
typescript
// home.fixture.ts
import { test as base } from '@playwright/test';
import { HomePage } from '../../pages/HomePage';

type HomeFixture = {
  homePage: HomePage;
};

export const test = base.extend<HomeFixture>({
  homePage: async ({ page }, use) => {
    const homePage = new HomePage(page);
    await homePage.goto();
    await use(homePage);   // ← THIS is what makes it a fixture
  }
});
```

The only thing that makes it a "fixture" is `base.extend()` — that's the actual Playwright API. The filename could literally be `banana.ts` and it would work the same.

---

## The `base.extend()` Pattern Explained Simply

Think of it like this:
```
base.extend() = "I want to add something new that every test can use"
typescript
// Playwright gives you `page` by default
test('example', async ({ page }) => { ... })

// base.extend() lets you ADD your own things alongside `page`
test('example', async ({ page, homePage, testUser }) => { ... })
//                                ^^^^^^^^  ^^^^^^^^
//                                your custom fixtures
The use() Function — The Key Concept
Inside base.extend(), the structure is always:

typescript
myFixture: async ({ page }, use) => {
  // 1. SETUP   — runs before the test
  const something = new Something(page);

  // 2. use()   — test runs HERE, receives `something`
  await use(something);

  // 3. TEARDOWN — runs after the test
  await something.cleanup();
}
use() is just Playwright's way of saying "pause here, run the test, then come back". Everything before use() is setup, everything after is teardown. That's the entire concept.

Side by Side — With vs Without
typescript
// ❌ Without fixture — setup repeated in every test
test('search works', async ({ page }) => {
  const homePage = new HomePage(page);  // manual
  await homePage.goto();                // manual
  await homePage.searchFor('iPhone');
});

test('banner visible', async ({ page }) => {
  const homePage = new HomePage(page);  // repeated again
  await homePage.goto();                // repeated again
  await homePage.verifyBanner();
});
typescript
// ✅ With fixture — setup lives in one place
test('search works', async ({ homePage }) => {
  await homePage.searchFor('iPhone');   // homePage already ready
});

test('banner visible', async ({ homePage }) => {
  await homePage.verifyBanner();        // homePage already ready
});
Summary
Thing	What it actually is
.fixture.ts extension	Just a naming convention, regular .ts file
base.extend()	The real Playwright API that creates fixtures
use()	Splits setup / test / teardown in one function
The fixture name e.g. homePage	What gets injected into async ({ homePage })
The whole point is: move repetitive setup out of tests and into one place, so tests only contain what they're actually testing.

Great question! Yes, the .d has a very specific meaning.

.d.ts = Declaration File
The d stands for declaration.

It tells TypeScript:

"This file only DESCRIBES types/shapes.
 It contains zero real/runnable code."
Normal .ts vs Declaration .d.ts
typescript
// user.ts — REAL file, has actual running code
export class User {
  name: string;
  constructor(name: string) {
    this.name = name;   // ← real logic, compiles to JavaScript
  }
}
typescript
// user.d.ts — DECLARATION file, only describes shape
export declare class User {
  name: string;        // ← no logic, never compiles to JavaScript
}
.d.ts files never produce any JavaScript output. They exist purely to tell TypeScript what shape something has.

Your env.d.ts Specifically
typescript
// src/types/env.d.ts

declare namespace NodeJS {
  interface ProcessEnv {
    BASE_URL: string;
    ADMIN_EMAIL: string;
    ADMIN_PASSWORD: string;
  }
}
Without this file:

typescript
process.env.ADMIN_EMAIL   // TypeScript sees: string | undefined
process.env.ADMINE_MAIL   // ← typo! TypeScript says nothing, no error
With this file:

typescript
process.env.ADMIN_EMAIL   // TypeScript sees: string ✅
process.env.ADMINE_MAIL   // ← TypeScript throws error immediately ✅
The file adds zero runtime behaviour. It just teaches TypeScript what process.env contains in your project.

Where .d.ts Files Come From in Real Projects
You've actually been using .d.ts files all along without realising it. When you install any npm package:

bash
npm install @playwright/test
```

Inside `node_modules/@playwright/test/` there are hundreds of `.d.ts` files:
```
node_modules/
  @playwright/
    test/
      index.d.ts     ← describes Page, Locator, expect, test etc.
That's how TypeScript knows what page.click(), page.fill(), expect() etc. look like — it reads those declaration files.

Three Common .d.ts Patterns You'll See
typescript
// 1. env.d.ts — extending existing types (your case)
declare namespace NodeJS {
  interface ProcessEnv {
    MY_KEY: string;
  }
}

// 2. global.d.ts — declaring global variables
declare const __APP_VERSION__: string;

// 3. module.d.ts — describing a JS library that has no types
declare module 'some-old-js-library' {
  export function doSomething(value: string): void;
}

Extension	Contains	Compiles to JS?
.ts	Real code + types	✅ Yes
.d.ts	Types/shapes only	❌ Never

The Key: dotenv Only Fills Gaps
This is the most important thing to understand:

typescript
// playwright.config.ts
dotenv.config({ path: `data/env/.env.${testEnv}` });
```

`dotenv.config()` has one rule:
```
"If a variable is ALREADY in process.env → leave it alone
 If a variable is MISSING             → fill it from .env file"
```

So on CI/CD:
```
GitHub Actions injects COMPANY_EMAIL into process.env FIRST
       │
       ▼
dotenv.config() runs → sees COMPANY_EMAIL already exists → skips it
       │
       ▼
process.env.COMPANY_EMAIL = GitHub Secret value  ✅
```

On local:
```
process.env has no COMPANY_EMAIL yet
       │
       ▼
dotenv.config() runs → COMPANY_EMAIL missing → fills from .env.stage
       │
       ▼
process.env.COMPANY_EMAIL = value from .env.stage  ✅
They never clash. Ever.

Great question! Let me explain everything from scratch.

What is a Pipeline / Workflow File?
Think of it like this:

You write test scripts → Playwright runs them on YOUR machine

But what about running them on GitHub's machine?
GitHub has its own computers (called "runners") sitting in the cloud.
You need to tell those computers:
  - what to install
  - what commands to run
  - what secrets to use

That instruction file is called a WORKFLOW file.
It lives in .github/workflows/
pipeline.yml vs playwright.yml — What's the Difference?
Absolutely nothing technically.

Both are just YAML files that define a GitHub workflow.
The NAME is just a label — you can call it anything:

  pipeline.yml        ← we named it this
  playwright.yml      ← GitHub auto-generates this when you
                         set up Playwright via their UI
  ci.yml              ← some teams call it this
  run-tests.yml       ← or this

They are ALL the same thing.
Just different names for the same concept.
Since GitHub may have auto-created playwright.yml when you set up the repo — you can delete it and keep only our pipeline.yml. No need for both.

YAML Basics First
Before reading the file, understand YAML syntax:

yaml
# This is a comment

name: John          # key: value

address:            # key with nested values (indent with 2 spaces)
  city: Mumbai
  pin: 411001

hobbies:            # key with a list
  - cricket
  - coding
  - reading

isActive: true      # boolean
age: 25             # number
That's literally all the YAML you need to read a pipeline file.

Now The Full Pipeline File — Every Line Explained
yaml
# ─────────────────────────────────────────────────────────────
# FILE: .github/workflows/pipeline.yml
#
# This file tells GitHub:
# "When someone triggers this manually,
#  spin up a computer, install everything,
#  and run our Playwright tests"
# ─────────────────────────────────────────────────────────────


# The name that appears in GitHub Actions UI
# GitHub Actions tab → you'll see this name in the list
name: WealthVest Playwright Tests


# ── TRIGGER ──────────────────────────────────────────────────
# This section answers: "WHEN should this pipeline run?"
on:

  # workflow_dispatch means: ONLY run when someone
  # manually clicks "Run workflow" in GitHub Actions UI
  # This is what you wanted — no automatic runs on push
  workflow_dispatch:

    # inputs = the dropdown options you see before clicking Run
    inputs:

      environment:
        description: 'Which environment to test?'
        required: true              # must select before running
        default: 'uat'              # pre-selected value
        type: choice
        options:
          - uat                     # option 1
          - stage                   # option 2 (future)
          - preprod                 # option 3 (future)

      test_suite:
        description: 'Which test suite to run?'
        required: true
        default: 'all'
        type: choice
        options:
          - all                     # run everything
          - smoke                   # only @smoke tagged tests
          - e2e                     # only tests/e2e/ folder
          - auth                    # only tests/auth/ folder
          - api                     # only tests/api/ folder

      browser:
        description: 'Which browser?'
        required: true
        default: 'chromium'
        type: choice
        options:
          - chromium
          - firefox


# ── JOBS ─────────────────────────────────────────────────────
# A job = a set of steps that run on ONE GitHub computer
# You can have multiple jobs — they can run in parallel
jobs:

  # This is the job name — just a label
  # You can name it anything
  playwright-tests:

    # runs-on = what kind of computer GitHub should use
    # ubuntu-latest = Linux machine (free, fast)
    # This is why we don't need cross-env in CI —
    # Linux supports env vars natively
    runs-on: ubuntu-latest

    # Give this job a readable name in the UI
    # ${{ }} = how you read variables in GitHub Actions
    # github.event.inputs.environment = what user selected in dropdown
    name: '${{ github.event.inputs.test_suite }} on ${{ github.event.inputs.environment }}'

    # ── STEPS ──────────────────────────────────────────────
    # Steps run ONE BY ONE, top to bottom
    # If any step fails → job stops (unless you say otherwise)
    steps:

      # STEP 1: Download your code onto GitHub's computer
      # Without this, the runner has no idea what your project is
      # actions/checkout@v4 is a pre-built action by GitHub
      - name: Checkout repository
        uses: actions/checkout@v4


      # STEP 2: Install Node.js on GitHub's computer
      # GitHub's machine is blank — nothing is installed
      # We need Node because Playwright runs on Node
      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'        # Node version to install
          cache: 'npm'              # cache node_modules between runs
                                    # makes subsequent runs faster


      # STEP 3: Install all npm packages
      # npm ci = reads package-lock.json exactly
      #        = installs same versions every time
      #        = faster and safer than npm install for CI
      # This installs: playwright, cross-env, dotenv, typescript etc.
      - name: Install dependencies
        run: npm ci


      # STEP 4: Install actual browser binaries
      # npm ci installs the Playwright LIBRARY
      # but the actual Chrome/Firefox browsers are separate
      # --with-deps also installs Linux system dependencies
      - name: Install Playwright browsers
        run: npx playwright install --with-deps ${{ github.event.inputs.browser }}


      # STEP 5: Run the tests
      # This is the main step
      - name: Run Playwright tests
        
        # env: injects variables into process.env for this step
        # ${{ secrets.SECRET_NAME }} reads from GitHub Secrets
        # (Settings → Secrets and variables → Actions)
        # These values are NEVER printed in logs — GitHub masks them
        env:
          # Tell playwright.config.ts which env to use
          TEST_ENV: ${{ github.event.inputs.environment }}

          # ── Credentials from GitHub Secrets ──────────────
          # Left side  = process.env variable name (what code reads)
          # Right side = GitHub Secret name (where value comes from)
          COMPANY_EMAIL:       ${{ secrets.COMPANY_EMAIL }}
          COMPANY_PASSWORD:    ${{ secrets.COMPANY_PASSWORD }}
          TRUST_EMAIL:         ${{ secrets.TRUST_EMAIL }}
          TRUST_PASSWORD:      ${{ secrets.TRUST_PASSWORD }}
          INDIVIDUAL_EMAIL:    ${{ secrets.INDIVIDUAL_EMAIL }}
          INDIVIDUAL_PASSWORD: ${{ secrets.INDIVIDUAL_PASSWORD }}
          API_KEY:             ${{ secrets.API_KEY }}
          BASE_URL:            ${{ secrets.BASE_URL_UAT }}
          API_BASE_URL:        ${{ secrets.API_BASE_URL_UAT }}

        # The actual command to run
        # We build it dynamically based on what user selected
        run: |
          if [ "${{ github.event.inputs.test_suite }}" = "all" ]; then
            npx playwright test --project=${{ github.event.inputs.browser }}

          elif [ "${{ github.event.inputs.test_suite }}" = "smoke" ]; then
            npx playwright test --grep @smoke --project=${{ github.event.inputs.browser }}

          elif [ "${{ github.event.inputs.test_suite }}" = "e2e" ]; then
            npx playwright test tests/e2e/ --project=${{ github.event.inputs.browser }}

          elif [ "${{ github.event.inputs.test_suite }}" = "auth" ]; then
            npx playwright test tests/auth/ --project=${{ github.event.inputs.browser }}

          elif [ "${{ github.event.inputs.test_suite }}" = "api" ]; then
            npx playwright test tests/api/ --project=${{ github.event.inputs.browser }}
          fi


      # STEP 6: Upload HTML report as artifact
      # Artifact = file you can download from GitHub Actions UI
      # if: always() = upload EVEN if tests failed
      #              = you need the report most when tests fail!
      - name: Upload HTML report
        uses: actions/upload-artifact@v4
        if: always()
        with:
          # Name of the downloadable artifact in GitHub UI
          name: playwright-report-${{ github.event.inputs.environment }}-${{ github.run_number }}
          # Which folder to upload
          path: reports/html/

workflow	The entire file — one automated process

inputs	Dropdowns you see before clicking Run

steps	Individual commands, run top to bottom

run:	Run a shell command directly
env:	Inject variables into that step

artifact	File you can download after the run
${{ }}	How you read variables inside the YAML file
